# Simple Comment Viewer Design Memo

このメモは、NDGRで取得できるコメント/運営イベントをどう扱うかの設計メモ。
ユーザーの説明を流さず、実装前の判断材料として残す。

## 基本方針

- 取得できるイベントはすべてDBに保存する。
- raw payload と、表示/処理用に正規化した値の両方を持つ。
- 設計の主役は「イベント種類」だけではなく、「アカウントIDごとの演出プロファイル」。
- 音声ファイル再生とVOICEVOX読み上げは別物として扱う。

## アカウントIDごとの個別設定

アカウントIDをDBに登録できるようにする。

登録したアカウントIDごとに、以下を設定できるようにする。

- スキン
- フォント
- ボイス/VOICEVOX系の読み上げ設定

このアカウントIDのユーザーがコメントした場合、そのユーザー専用の設定で表示/再生する。

## OBS向けコメント表示

スキンとフォントは、OBSへ送るコメント表示に使う。

OBSへ送信するものはHTMLにする。

OBS Browser Sourceで使えるように、HTMLの背景は透明にする。

表示イメージ:

- 横長のスキン
- 横長のフォント表示
- スキンの初期サイズは幅512px、高さ32pxを想定する。
- スキンの高さ/横幅は設定で変更できるようにする。
- コメントは右から左へ流れる
- 右端から出る時は最高速で入り、左端に近づくほど減速する。
- 移動は一定速度ではなく、減速イージングを使う。
- 20秒ごとに一段上へ移動する
- 新しいコメントが来た時も既存コメントは一段上へ移動する
- 新しいコメントは下段に出る

つまり、横スクロールするコメントレーンを複数段で管理する。

## 運営コマンドの表示と音声

運営コメント/運営コマンド系は、多くの場合、音声ファイルと一緒に流す。

基本動作:

- スキンとフォントをOBS HTMLへ流す。
- それと同時に、event_kindごとの音声ファイルを再生する。
- コメント本文が含まれる場合は、音声ファイルの再生中にVOICEVOX等で読み上げる。
- 音声ファイルが鳴り終わるまで読み上げ開始を待たない。

待たない理由:

- SE終了待ちにするとコメントがどんどん溜まる。
- コメント処理キューが詰まり、表示/読み上げの遅延が大きくなる。

つまり、運営コマンドでは以下を同時に走らせる。

```text
OBS表示: スキン + フォント + 変換済みテキスト
固定音声: event_kindごとのwav/mp3
読み上げ: コメント本文がある場合はSE再生中でも開始
```

## コメント処理キュー

コメントはすべて一度キューに入れる。

キューに入れたコメントを順次処理し、表示、VOICEVOX音声生成、音声再生、OBS送信へ流す。

重要:

- コメントは取りこぼさないように全件キューへ積む。
- 処理順序を管理する。
- 重い処理と表示処理を分離する。
- 表示するときは、コメント本文だけでなく、対象ユーザーのスキンとフォント設定も同時に流す。

## VOICEVOX処理

VOICEVOXは重いため、複数スレッド/ワーカーで処理する。

想定:

- VOICEVOX用ワーカーを3つ程度用意する。
- コメントはVOICEVOX生成待ちキューに入れる。
- 空いているワーカーが順次音声生成する。
- 生成後、再生キューへ渡す。
- 表示キューと再生キューの順序が破綻しないように管理する。

3つ程度のワーカーを用意する理由:

- 1本だけだとコメント速度に追いつかない可能性が高い。
- 複数コメントが短時間に来ても、音声生成待ちで詰まりにくくする。
- ただし、最終的に流す順序はキューで制御する。

処理イメージ:

```text
NDGR受信
  ↓
raw event DB保存
  ↓
comment/event queue
  ↓
アカウントID設定 or event_kind設定を解決
  ↓
OBS表示キューへ投入
  ↓
VOICEVOX生成キューへ投入
  ↓
VOICEVOX worker x 3
  ↓
音声再生キュー
```

## 読み上げ速度の自動調整

コメントが溜まってきたら、読み上げ速度を自動的に上げる。

目的:

- コメントが多い時に読み上げが延々と遅延しないようにする。
- キュー詰まりを緩和する。
- コメント量に応じて自然に処理速度を上げる。

設定例:

- 待機コメント 1個以上: 1.2倍
- 待機コメント 10個以上: 2.0倍
- 待機コメント 20個以上: 3.0倍

この段階設定はユーザーが変更できるようにする。

DB/設定としては、以下のような速度ルールを持つ。

```text
queue_size >= 1  -> speed_scale 1.2
queue_size >= 10 -> speed_scale 2.0
queue_size >= 20 -> speed_scale 3.0
```

VOICEVOXへ投げる直前に、現在の待機数から読み上げ速度を決める。

## 読み上げ前テキスト変換

VOICEVOXへ渡す前に、正規表現でコメント本文を変換できるようにする。

目的:

- URLをそのまま読ませない。
- `www` や `888` などを読みやすい表現に変換する。
- 読み上げに不要な文字列を省略/置換する。

例:

- `https?://\S+` -> `URL`
- `w+` -> `笑`
- `8{3,}` -> `拍手`

変換ルールは複数登録できるようにする。

各ルール:

- 有効/無効
- 正規表現パターン
- 置換後テキスト
- 適用対象
  - VOICEVOXのみ
  - OBS表示のみ
  - 両方

処理順序:

```text
raw comment
  ↓
DB保存
  ↓
表示用/読み上げ用に分岐
  ↓
読み上げ用テキストへ正規表現変換
  ↓
キュー待機数から速度決定
  ↓
VOICEVOX生成
```

## 運営コマンド系イベント

広告、ギフト、来場通知、ゲーム起動、その他コマンド系は、アカウントID設定とは別に、種類ごとの設定で扱う。

運営コマンド系では、event_kindごとに音声ファイルを決められるようにする。

例:

- nicoad: 広告用音声ファイル
- gift: ギフト用音声ファイル
- visitor: 来場通知用音声ファイル
- game: ゲーム起動/ゲーム通知用音声ファイル
- operator_comment: 運営コメント用音声ファイル
- system: その他システム通知用音声ファイル
- unknown: 未分類イベント用音声ファイル

## 運営コマンドの変換ルール

運営コマンドは、raw payloadをそのまま表示するのではなく、
「このコマンドが来たら、このように変換して出力する」というルールを持てるようにする。

例:

- 広告: `{advertiser_name}さんが{point}ptニコニ広告しました`
- ギフト: `{advertiser_name}さんが「{item_name}」を{point}ptギフトしました`
- 来場: `{name}さんが来場しました`

変換後テキストは、GUI表示、OBS表示、ログ表示、音声読み上げなどに使う。

## 分離して扱うもの

混ぜない。

- 固定音声ファイル: wav/mp3などのSE再生
- VOICEVOX/読み上げボイス: スキン/フォントと同じ演出プロファイル側
- スキン: OBS表示の見た目
- フォント: OBS表示の文字スタイル
- raw DB保存: すべてのイベントに対して必ず行う

## 現時点の設計軸

二本柱で考える。

1. アカウントID演出プロファイル
   - アカウントIDごとにスキン、フォント、ボイスを設定する。
   - コメントごとにその設定を適用する。

2. event_kind別コマンド処理プリセット
   - 広告、ギフト、来場通知などを種類ごとに扱う。
   - 種類ごとに固定音声ファイルと表示変換ルールを設定する。

## プロジェクト範囲

この設計と実装は、lab内のシンプルコメビュアプリだけで完結させる。

対象:

- `lab/simple_comment_viewer`

監視アプリ本体には手を出さない。

監視アプリ側の実装を参考にする場合も、直接改変せず、使えるGUIルールや部品の考え方だけをコメビュ側へ持ってくる。

## Git運用

このコメビュアプリは、アプリ内で独立したGit管理を行う。

ルール:

- `lab/simple_comment_viewer` の中で `git init` する。
- `.gitignore` を作ってからコミットする。
- コミット時は、このプロジェクトでも `git add .` を使う。
- コミットメッセージは日本語にする。

## ログ設計ルール

最初に共通ログ関数を作る。

ログレベルは5段階にする。

- TRACE
- DEBUG
- INFO
- WARN
- ERROR

ログ関数はアプリ全体で共通利用する。

ログを出す場所は絞る。

- 分岐
- 実行
- エラー
- 結果

この4種類の場面だけを基本ログ出力ポイントにする。

大量イベントごと、毎フレーム、細かすぎる内部状態を無制限に出すようなログは避ける。

## GUI基本ルール

監視アプリで使っていたGUIの基本ルールは、コメビュアプリにも持ってくる。

持ってくるもの:

- ウィンドウサイズを記憶する。
- テーブルのカラム幅を記憶する。
- テーブルのカラム順序を記憶する。
- テーブルのカラムはユーザーが移動できるようにする。
- テーブルのカラム幅はユーザーが変更できるようにする。
- 行の文字が長くても、ウィンドウ全体やカラム全体を勝手に巨大化させない。
- 横スクロール位置を更新で勝手に飛ばさない。
- 縦スクロール位置を更新で勝手に飛ばさない。
- ユーザーが選択中の行やコピー操作を、更新で壊さない。
- 新着行を追加しても、必要以上にフォーカスを奪わない。
- テーブルは行選択を基本にする。
- 右クリックメニューやコピー操作は、必要な表に入れる。

特に重要:

- 更新でジャンプしない。
- 長文でレイアウトを破壊しない。
- カラム状態を次回起動へ引き継ぐ。

これはコメビュでも最初から守る。
